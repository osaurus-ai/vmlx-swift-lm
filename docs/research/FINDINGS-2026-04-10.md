# Swift vs Python MLX Performance Gap — All Findings

**Date:** 2026-04-10
**Hardware:** M4 Max 128GB, macOS 26.3
**Model:** Qwen3.5-35B-A3B-4bit (MoE, 35B total / 3B active)
**Branch:** `fix/swift-compile-performance`

## The Question

Swift mlx-swift-lm gets 44 tok/s. Python mlx_lm gets 123 tok/s. Inferencer.app gets 107 tok/s. All on the same machine, same model, same MLX 0.31.1 C++ backend. Why?

## What We Ruled Out

### NOT a hardware issue
Same M4 Max. User verified Inferencer running locally on this machine. Same GPU, same kernels.

### NOT the MLX C++ version
Our MLX 0.31.1. Inferencer 0.24.2. Python mlx_lm 0.31.1 (same as us) — still 3x faster.

### NOT the mlx-swift compile() wrapper
Task 1 agent optimized it clean. ~0.1ms delta. Not the bottleneck.

### NOT the FFI bridge per-op overhead
Graph build is 1.66ms in Swift vs 1.73ms in Python — **Swift is faster**. FFI cost per op is not the issue.

### NOT the MambaCache subscript issue
Task 2 fixed it with _updateInternal. Uncompiled Qwen3.5 baseline unchanged at ~42 tok/s.

### NOT compile() kernel fusion
Python gets 123 tok/s WITHOUT compile. When we DO compile in Swift, it gives +13% on Qwen3.5 via TokenIterator path.

### NOT extra asType/bfloat16 ops
Grep shows only a comment, no runtime conversions in the decode hot path.

### NOT the eval() wrapper collection overhead
eval(token only) and eval(token + 80 cache arrs) both = 22.4ms. Cache arrays are already part of the token's graph.

## The Remaining Mystery

Python and Swift both call mlx_eval(vector_array) — the same C function. Swift's graph is same size as Python's. Same hardware. Same Metal kernels. Yet Swift takes 14ms longer inside mlx_eval.

This cannot be explained by:
- Swift wrapper code (profiled, trivial)
- Graph size (same)
- FFI crossings (graph build identical)
- Metal kernel version (same metallib)

Must be one of:
1. **Different stream / command queue scheduling** — Python mx.stream(generation_stream) creates isolated queue; Swift uses default.
2. **Different eval order** — Python may evaluate [logits, *cache_state] in different order enabling Metal batching.
3. **Init-time Metal state** — Python sets a property at import we don't.
4. **Evaluatable/Updatable protocol dispatch overhead** — 80 cache state arrays through protocol witness table per eval.

## UPDATE 2026-04-10 Late Afternoon: Deep Dive Findings

### Python stream comparison

- Python `mx.eval` on **default stream**: 13.70ms/step
- Python `mx.eval` on **generation_stream** (dedicated): 10.19ms/step
- Delta: 3.5ms benefit from dedicated stream

### Swift stream experiment

Tried `MLX.Stream.setDefault(Stream(Device.defaultDevice()))` before the benchmark loop:
- **Result: 27.2ms/step** — WORSE than default stream (22.7ms)
- Dedicated stream hurt performance in Swift

This is opposite to Python's behavior. Suggests Swift's stream switching has extra overhead OR the cache state writes are bound to a different stream than expected.

### Still unexplained: 8.8ms gap on default stream

- Python default stream: 13.70ms
- Swift default stream: 22.50ms
- Gap: 8.80ms

Both call `mlx_eval(vector_array)` → `mlx::core::eval(std::vector)`. Same C++ function. Same inputs. Same cache state count. Same graph structure.

### MTL4 and Metal 3 — both have the framework
Inferencer and our binary both link MTL4CommandQueue/MTL4CommandEncoder from system frameworks, but MLX C++ does NOT use MTL4 — it uses the older Metal 3 command queue API (verified in `.build/checkouts/mlx-swift/Source/Cmlx/mlx/mlx/backend/metal/device.cpp`).

### Command buffer tuning experiment

MLX has env vars `MLX_MAX_OPS_PER_BUFFER` and `MLX_MAX_MB_PER_BUFFER`. Defaults on M4 Max (arch='s'): ops=50, mb=50.

Tested `MLX_MAX_OPS_PER_BUFFER=500 MLX_MAX_MB_PER_BUFFER=500` → **GPU HANG**.
Tested `=100/=100` → GPU HANG.
Tested `=80/=80` → works, ~same speed.

Larger command buffers cause GPU hang. Default values are already optimal.

### What Python's eval binding does

```cpp
m.def("eval", [](const nb::args& args) {
    std::vector<mx::array> arrays = tree_flatten(args, false);
    { nb::gil_scoped_release nogil; eval(arrays); }
});
```

Key detail: **Python releases the GIL** during eval. Swift has no GIL so this isn't directly applicable, but it proves Python's binding is extremely thin.

### What Swift's eval wrapper does

```swift
public func eval(_ arrays: some Collection<MLXArray>) {
    let vector_array = new_mlx_vector_array(arrays)  // allocs Swift Array of ctx pointers + C vector
    mlx_eval(vector_array)                           // calls mlx::core::eval
    mlx_vector_array_free(vector_array)              // frees C vector
}
```

Swift's wrapper is also thin. The `arrays.map { $0.ctx }` allocation is the only Swift-side overhead and was measured at ~0.004ms for 80 arrays.

### The mystery deepens

Both Python and Swift:
1. Build a C vector of array handles
2. Call the same `mlx::core::eval` C++ function
3. Same input count, same graph, same Metal kernels

Yet the C++ function takes 13.7ms in Python and 22.5ms in Swift **on the default stream**, both doing exactly the same work.

### Leading hypothesis: static linking difference

- Python uses `libmlx.dylib` (dynamically linked)
- Swift uses statically linked `libmlx-core.a` via Cmlx target
- Dynamic loading might use different TLS, different memory layout, or different inlining

The C++ `eval()` function's thread-local scheduler state could be faster to access in a dylib than in a static lib due to different `__thread` / `thread_local` codegen.

This is hard to verify without relinking. Would need to build Swift TestRunner against the dylib.

### Alternative hypothesis: graph flattening differs

Python `tree_flatten(args, false)` might produce a different array count than Swift's `[ny] + cacheState`. Python's result is unique arrays after tree walk. Swift's is the literal concatenated Collection.

If the cache state arrays include duplicates (e.g., both `keys` and `values` point to the same underlying buffer after prefill), Python's tree_flatten might dedup them while Swift's literal list wouldn't.

Need to verify by counting unique arrays in Python's tree_flatten output.

## FINAL CONCLUSIONS (2026-04-10)

### What we tried

1. **Task 1: Optimize compile() wrapper** (`.build/checkouts/mlx-swift/Source/MLX/Transforms+Compile.swift`)
   - Cached `stateInputs`/`stateOutputs` as fields (computed once in init)
   - Removed instance NSLock around `innerCall`
   - Removed `evalLock` from compile path
   - **Result: ~0.1ms improvement, not the bottleneck**

2. **Task 2: ArraysCache `_updateInternal`** (`Libraries/MLXLMCommon/KVCache.swift`)
   - Subscript setter now uses `existing._updateInternal(newValue)` when slot is populated
   - Preserves reference identity for compile() tracing
   - **Result: Makes MambaCache compile-compatible. Uncompiled baseline unchanged.**

3. **Task 3: Heterogeneous cache compile** (`Libraries/MLXLMCommon/Evaluate.swift`)
   - `setupCompiledDecode` now allows `ArraysCache` through the guard
   - Converts only `KVCacheSimple` → `CompilableKVCache`
   - **Result: +14% Qwen3.5 via TokenIterator (39.1 → 44.5 tok/s). Llama: +36% IDENTICAL output.**

4. **Dedicated generation stream** (`Stream.setDefault`)
   - **Result: WORSE in Swift (27ms vs 22.7ms default). Opposite of Python.**

5. **Larger command buffers** (`MLX_MAX_OPS_PER_BUFFER=500`)
   - **Result: GPU HANG. Current defaults are optimal.**

6. **Recompile Cmlx with -O3 -DNDEBUG -fomit-frame-pointer**
   - **Result: WORSE (46ms vs 24ms). SPM's default optimization is already optimal.**

7. **Binary comparison Inferencer vs ours**
   - Same metallib (byte-identical)
   - Same MLX C++ primitives and kernels
   - Same MTL4 framework linking (neither uses MTL4 compute APIs — both use Metal 3)
   - Inferencer-specific: `ChunkedKVCache`, `CachedModelWrapper`, `XLLMModelFactory`, `fragmentPreloadedLibraries`, `WiredMemoryManager`

### What we could NOT explain

**The 8.8ms gap between Python default-stream eval (13.70ms) and Swift default-stream eval (22.50ms).**

Both call `mlx_eval(vector_array)` → `mlx::core::eval(std::vector)`. Same C++ function. Same graph. Same inputs. Same Metal kernels. Yet Swift is consistently ~8ms slower per step.

### What Inferencer likely does that matters

Based on binary strings and symbol analysis, the most plausible hypotheses:

1. **Fragment preloading** via `setFragmentPreloadedLibraries:` — JIT-compiles Metal shader fragments upfront. Affects cold-start; unclear if it affects per-step after warmup.

2. **CachedModelWrapper persists state between requests** — amortizes setup cost across multiple inferences. Not relevant for single per-step measurement.

3. **ChunkedKVCache with `startPosition` property** — similar to our CompilableKVCache but simpler. Might use chunked updates that play better with compile().

4. **WiredMemoryManager actively used** — we have the class but don't wire it into TestRunner. Needs `sudo sysctl iogpu.wired_limit_mb=<N>` + `WiredMemoryTicket` integration in generation loop.

5. **Custom fork of mlx-swift** — Inferencer builds from `/Users/x/Dev/projects/inferencer/inferencer-macos/Dependencies/mlx-swift/` which is NOT a public fork. They may have C++ patches or Swift shortcuts not in upstream.

### Recommended next steps

If continuing to chase the 2.6x gap:

1. **Link Swift against libmlx.dylib** instead of static Cmlx. Use the exact same build Python uses. If that closes the gap, it's a static-vs-dynamic linking artifact.

2. **Instrument with Xcode Instruments → Metal System Trace**. See actual GPU kernel timing and Metal command buffer submission overhead. Compare Python and Swift runs directly.

3. **Hook `mlx::core::eval` with dyld injection** to trace exact std::vector contents on both sides. Verify the inputs are truly identical.

4. **Try linking against the exact Python libmlx.dylib** in a Swift test target using `@rpath` override.

5. **Port Inferencer's ChunkedKVCache approach** — their cache might have a structure we're missing that reduces graph op count.

6. **Port `fragmentPreloadedLibraries` call** at model load time. See if first-token latency improves even if per-step doesn't.

### What's committed and ready to ship

- `7920bb0` in mlx-swift checkout: compile wrapper optimization
- `4c026ad` in mlx-swift-lm: ArraysCache subscript `_updateInternal`
- `cc50aac` in mlx-swift-lm: heterogeneous cache compile enabled

These three together give measurable improvements on all models where compile is applicable:
- Llama 3.2 1B: +26-41% (IDENTICAL output)
- Qwen3.5 35B via TokenIterator: +14%
- No regressions on any tested model

### The uncomfortable truth

We identified the gap (~8.8ms per step in eval), ruled out the obvious culprits (wrapper, cache type, graph size, op count, stream, buffer size, opt level), and still don't have a concrete code change that closes it.

The gap is real and measurable, but the root cause is hidden in either:
- A subtle C++ runtime difference (static vs dynamic linking, thread_local storage, std::vector construction)
- An initialization state difference between Python's mx module import and Swift's MLX lazy init
- A Metal driver interaction that differs based on how the process was started (Python dylib vs Swift statically linked binary)

Closing this gap likely requires either:
1. Instrumented profiling (Metal System Trace)
2. Relinking Swift against libmlx.dylib for A/B test
3. Access to Inferencer's source code or the mlx-swift maintainer's help

Neither of these is a quick fix. The Task 1-3 changes are the best shippable wins from this session.

### What we verified Swift has
- Matching `MLXFast.metalKernel` for gated_delta (GatedDelta.swift:108)
- Matching `compiledSwiGLU` (SwitchLayers.swift:28) 
- Matching `computeGatedDeltaG` (GatedDelta.swift:17)
- Matching `compiledSigmoidMultiply` (Qwen3Next.swift:17)

### What Swift was missing
- `_compiledPreciseSwiGLU` — Python's `_precise_swiglu` which fuses silu(fp32) + cast + mul + cast into 1 op. **Added to Qwen3Next.swift.**

### Result of adding _compiledPreciseSwiGLU
- Eval only: 22.5 → 22.5-23.1ms (within noise)
- Uncompiled: 24.4 → 24.2ms (within noise)
- **No measurable improvement.**

### Why the fix didn't work

Hypothesis: `compile(shapeless: true)` in Swift produces the same set of Metal kernel dispatches as uncompiled code — it just caches the graph trace. Python's `@mx.compile` apparently produces an actual FUSED Metal kernel (a single `Compiled*` primitive in the graph). Swift's compile either:
1. Doesn't fuse at the C++ level the same way
2. Fuses but the fused ops don't have the same "one Metal dispatch = multiple source ops" behavior
3. Is fusing but per-dispatch is already fast enough that we can't measure the savings

### What would definitively prove this

Dump the Swift graph as .dot (we attempted this via a custom `mlx_dump_graph_to_file` C bridge but ran into SPM rebuild issues — the new .cpp file in `mlx-generated/` wasn't picked up by the Cmlx build target).

If we could dump it, we'd see whether Swift has `Compiled*` fused primitives or just raw primitives like Multiply, Sigmoid, AsType individually.

## FINAL STATUS

The performance gap between Swift (22.5ms/step) and Python (10ms/step) on Qwen3.5-35B-A3B-4bit remains **not fully explained and not fully closed** despite exhaustive investigation:

- Graph build time is IDENTICAL between Python and Swift
- Individual kernel dispatch latency is IDENTICAL (matmul 4096: 9.4ms both; small add 64x64: Swift actually faster at 117us vs Python 145us)
- Metal kernels are byte-identical (same metallib)
- MLX C++ version is the same (0.31.1)
- Compile wrapper overhead was minimized (Task 1 optimizations applied)
- Cache types are compile-compatible (Task 2)
- Compile activates on Qwen3.5 (Task 3)
- Matching `_precise_swiglu` fusion added (new work)

The only remaining explanation consistent with the data: **Python's `mx.compile` produces fused Metal kernel primitives at the C++ level** (visible as `CompiledSigmoidMultiplyMultiply` etc. in the graph), while Swift's `compile()` may not produce the same fused primitives — OR it does but the benefit gets lost somewhere in our code path.

The user's frustration is warranted: we have ruled out every obvious explanation, made targeted fixes matching Python's approach, and still see ~22ms vs Python's 10ms.
